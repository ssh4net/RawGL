# RawGL Documentation Strategy

This note captures the documentation direction for RawGL after reviewing the
current documentation stacks used by OpenImageIO and OpenMeta.

The main conclusion is simple:

- use **OpenImageIO** as the target for content quality and user experience
- use **OpenMeta** as the closer reference for CMake wiring and GitHub Pages automation

RawGL should not try to publish its current planning notes directly as the user
documentation site. Those notes are still useful, but they are engineering
documents, not product documentation.

## 1. Reference Summary

### 1.1 OpenImageIO

OpenImageIO has a rich user-facing documentation site built from:

- hand-written narrative pages under `src/doc/`
- Doxygen XML generated from public headers
- Sphinx + Breathe for the final site
- Furo theme
- Read the Docs publishing

Notable strengths:

- strong landing page and navigation structure
- conceptual introductions before API details
- large amount of practical user documentation for both C++ and Python
- dedicated documentation for CLI tools
- glossary/appendix material
- diagrams, figures, screenshots, and example outputs
- command-line help converted into man pages as part of the docs build

The main lesson from OIIO is not just the tool choice. It is the content
strategy:

- explain the mental model first
- document tasks and workflows, not only symbols
- include real examples
- make the docs readable without forcing the reader through the API reference

### 1.2 OpenMeta

OpenMeta uses a smaller but cleaner stack that is closer to what RawGL should
copy in the near term:

- `docs/Doxyfile`
- `docs/sphinx/conf.py.in`
- Sphinx + Breathe
- CMake options for Doxygen and Sphinx builds
- GitHub Pages workflow that builds and publishes the HTML site
- install-time docs generation through CMake scripts

Notable strengths:

- docs are opt-in and explicit in CMake
- out-of-source Sphinx configuration via `conf.py.in`
- simple API reference page generated by Breathe
- clear separation between:
  - engineering notes in Markdown
  - published Sphinx site
  - generated Doxygen XML

The main lesson from OpenMeta is implementation discipline:

- keep docs generation integrated into CMake
- fail early when Doxygen/Sphinx dependencies are missing
- make CI/publishing straightforward

## 2. RawGL Current State

RawGL currently has:

- several useful engineering notes under `docs/`
- incremental Doxygen comments on the public headers
- no dedicated Doxygen configuration
- no Sphinx site
- no docs-specific CMake targets
- no docs CI/publishing workflow

This means RawGL already has documentation content, but almost all of it is
internal planning content rather than user documentation.

## 3. Recommended Direction

### 3.1 Overall approach

RawGL should adopt:

- **OpenMeta-style implementation**
- **OpenImageIO-style content quality**

That means:

1. build the site with Doxygen XML + Sphinx + Breathe
2. keep public API comments in headers as the API truth source
3. write narrative user docs separately from the API reference
4. publish a readable HTML site through GitHub Pages

### 3.2 Documentation tiers

RawGL should separate documentation into three tiers.

#### A. Published user documentation

This is the future site and should cover:

- what RawGL is
- how to install/build it
- quick start
- CLI workflows
- Python workflows
- C++ facade usage
- multipass workflows
- IO and batch model
- examples and limitations
- API reference

This should live under a new docs-site subtree, for example:

- `docs/sphinx/`

#### B. Generated API reference

This should come from:

- Doxygen comments in `include/rawgl/*.h`
- Doxygen XML consumed by Breathe

This should document the stable public surface first:

- `rawgl.h`
- `rawgl_io.h`
- `rawgl_batch.h`
- `rawgl_cli.h`
- `rawgl_core.h` only while transitional API is still intentionally public

#### C. Engineering notes

The current planning notes should stay in `docs/`, but they should not become
the public site navigation by default.

These notes are still valuable for development:

- refactor plan
- module split
- batch design
- public API redesign
- type-support staging

But they are a different document class.

## 4. Authoring Format Recommendation

OpenImageIO uses reStructuredText for its whole site. OpenMeta uses RST for the
site and Markdown for engineering notes.

For RawGL, the pragmatic first-wave choice is:

- use **Sphinx**
- use **reStructuredText** for the initial published site
- keep Markdown for engineering notes under `docs/`

Rationale:

- this matches both OpenImageIO and OpenMeta more closely
- it removes one extra Python dependency from the first-wave docs build
- Breathe examples and Sphinx cross-reference examples are simpler to follow in RST

If RawGL later wants MyST for narrative pages, that can still be added without
changing the overall Doxygen + Sphinx + Breathe structure.

## 5. Proposed Site Structure

The first published documentation site should be intentionally compact and
high-value.

### 5.1 Landing structure

- `index`
- `quick_start`
- `build_and_install`
- `concepts`
- `cli`
- `python`
- `cpp`
- `multipass_workflows`
- `io`
- `batch`
- `examples`
- `limits_and_caveats`
- `api`

### 5.2 Content intent

#### `quick_start`

Minimum working examples:

- one CLI fragment pass
- one Python `image(...)` example
- one C++ `Session -> Workflow -> PreparedWorkflow` example

#### `concepts`

Explain the model in plain language:

- `Session`
- `Workflow`
- `PreparedWorkflow`
- `RunSettings`
- `RunResult`
- `IoRuntime`
- `BatchRunner`

This should be the RawGL equivalent of OIIO's introductory conceptual docs.

#### `cli`

Document:

- pass types
- inputs/outputs
- file-oriented workflows
- common examples

#### `python`

Document the high-level Python surface first:

- `image(...)`
- `render(...)`
- `compute(...)`
- `prepare_*`
- NumPy input/output

Then document the advanced layer separately:

- multipass builder
- explicit pass objects
- batch bindings

#### `cpp`

Focus on the public facade, not internals.

#### `examples`

This is where RawGL can get closer to OIIO quality over time:

- show real examples with short explanations
- use generated images or diagrams where they add clarity
- link directly to repo examples

## 6. Build and Generation Model

RawGL should add:

- `docs/Doxyfile`
- `docs/requirements.txt`
- `docs/sphinx/conf.py.in`
- `docs/sphinx/index.md` or `index.rst`

And CMake options similar to OpenMeta:

- `RAWGL_BUILD_DOCS`
- `RAWGL_BUILD_SPHINX_DOCS`

Suggested targets:

- `rawgl_docs`
  - generate Doxygen XML and optionally HTML
- `rawgl_docs_sphinx`
  - depends on `rawgl_docs`
  - builds published HTML site

Suggested output layout:

- `build/docs/doxygen/`
- `build/docs/html/`

Suggested install behavior:

- install generated HTML docs under `${CMAKE_INSTALL_DOCDIR}`
- keep docs installation optional

## 7. CI and Publishing

RawGL should add a GitHub Actions workflow similar to OpenMeta:

- build on pull requests when docs inputs change
- deploy on `main`
- publish GitHub Pages artifact from `build/docs/html`

Suggested docs workflow inputs:

- `docs/**`
- `include/rawgl/**`
- `src/python/**` for Python docs
- `README.md`
- `CMakeLists.txt`
- docs workflow file itself

Suggested build dependencies in CI:

- Python 3.12
- `doxygen`
- `graphviz`
- `ninja-build`
- Python packages from `docs/requirements.txt`

## 8. What RawGL Should Copy From OIIO First

These are the highest-value OIIO traits to copy early:

1. strong introduction and conceptual orientation
2. user-facing examples on nearly every major page
3. separate pages for CLI, Python, and C++ usage
4. appendices/glossary/limitations once the basics exist
5. visual examples where outputs matter

These are lower priority for RawGL:

- man page generation from `--help`
- PDF/epub packaging
- large volume of generated figures

Those are useful, but not first-wave necessities.

## 9. First Implementation Wave

RawGL should implement the docs stack in this order.

### Phase 1: scaffolding

- add `docs/Doxyfile`
- add `docs/requirements.txt`
- add `docs/sphinx/conf.py.in`
- add `docs/sphinx/index.md`
- add `RAWGL_BUILD_DOCS`
- add `RAWGL_BUILD_SPHINX_DOCS`
- add `rawgl_docs` and `rawgl_docs_sphinx`

### Phase 2: minimum published content

- `quick_start`
- `concepts`
- `python`
- `cli`
- `api`

### Phase 3: CI

- add GitHub Pages docs workflow
- publish HTML site from `main`

### Phase 4: content expansion

- `cpp`
- `multipass_workflows`
- `io`
- `batch`
- `examples`
- `limits_and_caveats`

## 10. Practical Rule For RawGL

RawGL documentation should follow this rule:

- **planning notes stay in Markdown under `docs/`**
- **published user docs live in the Sphinx docs tree**
- **API reference comes from public headers**

That gives RawGL:

- OIIO-style user docs over time
- OpenMeta-style implementation discipline immediately
